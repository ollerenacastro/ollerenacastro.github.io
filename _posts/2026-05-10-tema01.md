## 1. Instalación del Blog Personal Gratuito con GitHub Pages y Jekyll ✍️ 

- Vídeo de referencia ([youtube tutorial](https://www.youtube.com/watch?v=m1RYsmOMPLs&t=801s)).


--------------------------------------------------------------------------------

Este tutorial describe cómo configurar y personalizar rápidamente un blog para desarrolladores utilizando el tema "Chirpy Starter", hospedaje gratuito en GitHub Pages, y la herramienta de generación de sitios estáticos Jekyll. A diferencia de guías genéricas, esta versión incluye la solución al problema de compatibilidad entre Ruby 4.x y Chirpy — un obstáculo frecuente en instalaciones recientes de Windows que no está documentado en el tutorial oficial.

### Paso 1: Configuración Inicial del Repositorio

1. **Acceder al Tema:** Diríjase al repositorio de la plantilla llamado **Chirpy Starter** ([github chirpy starter](https://github.com/cotes2020/chirpy-starter)).
2. **Crear Repositorio:** Haga clic en **"use template"** (usar plantilla).
3. **Nombrar el Repositorio:** Es crucial nombrar el nuevo repositorio usando su nombre de usuario de GitHub seguido de `.github.io`.
    *   *Ejemplo:* Si su usuario es `username`, el repositorio debe llamarse `username.github.io`.
4. **Crear:** Haga clic en **'create repository'** (crear repositorio).

### Paso 2: Configurar la URL Base

Una vez creado el repositorio:

1. **Editar Archivo de Configuración:** Vaya al archivo llamado `_config.yml`.
2. **Modificar URL:** Haga clic en editar y desplácese hacia la variable `URL`.
3. **Añadir URL Personal:** Agregue la URL completa de su blog, utilizando el formato `https://username.github.io`.
    *   *Ejemplo:* `https://ollerenacastro.github.io`.
4. **Guardar Cambios:** **Confirme los cambios** (commit). Puede añadir un mensaje de commit simple, como "adding personal username URL".

### Paso 3: Despliegue Automático en GitHub Pages

1. **Revisar Acciones:** Navegue a la pestaña **"GitHub actions"**.
2. **Proceso de Despliegue:** Verá que GitHub está realizando automáticamente la compilación y el despliegue del blog en GitHub Pages, el cual se aloja de forma gratuita. Este proceso tomará unos segundos.
3. **Verificar:** Una vez que las etapas de **'build'** (compilación) y **'deploy'** (despliegue) se completen [2, 3], su plantilla estará activa en el enlace `username.github.io`.
    *   *Nota:* Esta plantilla incluye características útiles como categorías, etiquetas, archivos, y un tema oscuro que se puede cambiar.

### Paso 4: Clonación Local y Preparación

Para personalizar el blog, necesitará trabajar localmente:

1. **Clonar el Repositorio:** Vaya a la sección **'code'** y copie el enlace de clonación.
2. **Abrir PowerShell:** Abra PowerShell y navegue a la carpeta donde desea guardar el proyecto.
3. **Ejecutar Clonación:** Ejecute:
    ```bash
    git clone [enlace_copiado]
    ```
4. **Abrir en Editor:** Navegue al directorio del repositorio (`cd [nombre_del_repositorio]`) y ábralo usando VS Code u otro editor para realizar cambios localmente.

### Paso 5: Instalación de Ruby y Jekyll en Windows

Jekyll es el motor que convierte los archivos Markdown en un sitio web. Para que funcione necesita Ruby, que es el lenguaje en que Jekyll está escrito. El tema Chirpy declara explícitamente que requiere **Ruby ~> 3.1**, es decir, cualquier versión 3.1, 3.2 o 3.3 — pero no Ruby 4.x, que introdujo cambios que aún rompen la compatibilidad.

#### 5.1 Desinstalar versión incompatible (si aplica)

Si ya tiene Ruby 4.x instalado, es necesario desinstalarlo antes. Ruby 4.0 eliminó y renombró varias funciones internas (`rb_thread_call_without_gvl`, `OBJ_TAINT`) que algunos gems aún usan. Intentar instalar los gems del proyecto con Ruby 4.x fallará con errores de compilación.

- Panel de Control → Programas → Desinstalar un programa
- Seleccionar **Ruby 4.0.3-1-x64** → Desinstalar

![Desinstalar Ruby 4.x desde Programas y características](/assets/images/tema01/ruby_08.png)

#### 5.2 Descargar Ruby+Devkit 3.3.11

Descargue el instalador desde [rubyinstaller.org/downloads](https://rubyinstaller.org/downloads/). En la sección **WITH DEVKIT**, elija `Ruby+Devkit 3.3.11-1 (x64)`.

> **¿Por qué WITH DEVKIT y no solo Ruby?** Varios gems (como los que manejan imágenes o conexiones nativas) contienen código en C que debe compilarse en su máquina durante la instalación. El Devkit incluye el compilador GCC y las herramientas MSYS2/MINGW necesarias para esa compilación. Sin él, `bundle install` fallaría al instalar esos gems.

![Seleccionar Ruby+Devkit 3.3.11-1 (x64) en la página de descargas](/assets/images/tema01/ruby_01.png)

#### 5.3 Ejecutar el instalador

Al abrir el instalador, Windows Defender puede mostrar una advertencia porque el ejecutable no tiene firma digital de una empresa grande. Es software legítimo y de código abierto — haga clic en **"Run anyway"**.

![Advertencia de Windows Defender — hacer clic en Run anyway](/assets/images/tema01/ruby_02.png)

Seleccione **"Install for me only (recommended)"**. Esta opción instala Ruby solo para su usuario, sin necesitar permisos de administrador. Es la opción recomendada en entornos de desarrollo personal.

![Seleccionar Install for me only](/assets/images/tema01/ruby_03.png)

Acepte la licencia y haga clic en **Next**.

![Aceptar licencia de Ruby](/assets/images/tema01/ruby_04.png)

Al finalizar, asegúrese de que **"Run 'ridk install'"** esté marcado antes de hacer clic en **Finish**. Esto lanzará automáticamente el siguiente paso.

![Finalizar con ridk install marcado](/assets/images/tema01/ruby_05.png)

#### 5.4 Instalar MSYS2 y toolchain de desarrollo

Se abrirá una consola de RubyInstaller2. Escriba `3` y presione Enter para instalar la opción **MSYS2 and MINGW development toolchain**.

> **¿Por qué la opción 3?** Esta instala tanto MSYS2 (entorno de compilación) como MINGW (compilador GCC para Windows). Las opciones 1 y 2 son subconjuntos — la 3 incluye todo lo necesario para compilar cualquier gem con extensiones nativas.

![Seleccionar opción 3 en RubyInstaller2](/assets/images/tema01/ruby_06.png)

Durante la instalación verá muchas líneas `gpg: removing stale lockfile` — esto es **completamente normal**. Son mensajes de limpieza del sistema de claves de GPG. Solo espere a que termine.

![Mensajes gpg normales durante la instalación](/assets/images/tema01/ruby_07.png)

> **Si los mensajes `gpg` persisten por varios minutos:** En algunos equipos este proceso puede extenderse indefinidamente sin avanzar. Si lleva más de 5 minutos sin cambiar, puede presionar `Ctrl+C` para interrumpirlo. Ruby ya quedó instalado correctamente en ese punto — el paso de MSYS2 es complementario y puede interrumpirse sin perder la instalación principal.

#### 5.5 Verificar la instalación

Abra un **nuevo PowerShell** — es importante que sea una ventana nueva, no la que tenía abierta. Esto es porque Windows carga las variables de entorno (como el PATH que apunta a la nueva instalación de Ruby) al iniciar la sesión. Una terminal abierta antes de la instalación no las recargará.

```powershell
ruby -v
```

Debe mostrar `ruby 3.3.11`. Luego instale Jekyll y Bundler:

```powershell
gem install jekyll bundler
```

> **¿Por qué instalar Jekyll globalmente si después usaremos `bundle install`?** `gem install` pone Jekyll disponible como comando en cualquier directorio. `bundler` es el gestor de dependencias que luego usará el archivo `Gemfile` del proyecto para instalar la versión exacta del tema y sus plugins. Son dos niveles distintos: uno global y uno por proyecto.

![Verificar ruby -v y ejecutar gem install](/assets/images/tema01/ruby_09.png)

![Salida de gem install jekyll bundler](/assets/images/tema01/ruby_10.png)

#### 5.6 Corregir el Gemfile del proyecto

El `Gemfile` del repositorio incluye el gem `wdm` (Windows Directory Monitor), que monitorea cambios de archivos en tiempo real. La versión disponible (0.1.1, del año 2014) usa funciones internas de Ruby que fueron eliminadas en Ruby 3.x, por lo que su compilación falla. Como Jekyll puede monitorear cambios sin `wdm`, simplemente lo eliminamos.

Abra el archivo `Gemfile` en la raíz del repositorio y reemplace su contenido por:

```ruby
# frozen_string_literal: true

source "https://rubygems.org"

gem "jekyll-theme-chirpy", "~> 7.1", ">= 7.1.1"

gem "html-proofer", "~> 5.0", group: :test

platforms :windows, :jruby do
  gem "tzinfo", ">= 1", "< 3"
  gem "tzinfo-data"
end
```

Guarde el archivo y commitee el cambio para que quede registrado en su repositorio:

```powershell
git add Gemfile
git commit -m "fix: remove wdm gem incompatible with Ruby 3.3"
git push
```

> **¿Por qué commitear este cambio?** Si no lo hace, la próxima vez que clone el repositorio en otra máquina o lo comparta, el `Gemfile` seguirá teniendo el error y el proceso de instalación fallará nuevamente.

#### 5.7 Instalar dependencias del proyecto

A diferencia de `gem install` (que instala globalmente), `bundle install` lee el `Gemfile` del proyecto e instala exactamente los gems que el proyecto necesita, en las versiones especificadas. Esto garantiza que todos los estudiantes tengan el mismo entorno.

```powershell
bundle install
```

Si todo está correcto verá: `Bundle complete! 4 Gemfile dependencies, 66 gems now installed.`

### Paso 6: Ejecutar y Personalizar el Blog Localmente

1. **Ejecutar Servidor Local:** Una vez instaladas las dependencias, ejecute el servidor local:
    ```powershell
    bundle exec jekyll serve
    ```
    Este comando construye el proyecto y lo ejecuta en su máquina.
2. **Visualizar:** El servidor se ejecutará en un enlace local (se puede acceder haciendo Control + clic en el enlace proporcionado en la terminal).
3. **Personalizar `_config.yml`:** Mientras el servidor se ejecuta, edite el archivo `_config.yml` para personalizar:
    *   **`title`:** Cambie el título (ejemplo: su nombre).
    *   **`tagline`:** Agregue un eslogan.
    *   **`description`:** Agregue una descripción (esto ayuda a que Google encuentre su blog).
    *   **Enlaces Sociales:** Agregue su nombre de usuario de GitHub y Twitter para que los enlaces se completen automáticamente.
    *   **`avatar`:** Busque la variable `Avatar` y pegue un enlace directo a su imagen de perfil (ejemplo: copiar el enlace de la imagen de perfil de Twitter).

### Paso 7: Crear la Primera Publicación

Los posts deben seguir un formato específico para el nombre del archivo:

1. **Convención de Nombres:** El nombre del archivo debe incluir la fecha (YYYY-MM-DD) y el nombre de la publicación, separados por guiones.
    *   *Ejemplo:* `2025-09-21-hello-world-post.md` (asumiendo que la fecha del post era el 21 de setiembre).
2. **Añadir Metadatos (Front Matter):** Al comienzo del archivo, añada la siguiente metadata obligatoria (utilizando el formato YAML):
    *   `title`
    *   `date`
    *   `categories`
    *   `tags`
3. **Escribir Contenido:** Añada su contenido (ejemplo: un encabezado "Hello world" seguido de texto).
4. **Verificar Localmente:** Guarde el archivo y actualice su servidor local para ver la nueva publicación.

### Paso 9: Subir Cambios a GitHub

Una vez que esté satisfecho con las personalizaciones y las publicaciones locales, suba los cambios al repositorio de GitHub para que aparezcan en su sitio web en vivo:

1. **Agregar Cambios:** Abra una nueva terminal y añada todos los cambios realizados:
    ```bash
    git add .
    ```
2. **Confirmar Cambios:** Añada un mensaje de commit:
    ```bash
    git commit -m "personalizando mi blog personal"
    ```
3. **Empujar a GitHub:** Suba los cambios al repositorio remoto:
    ```bash
    git push
    ```
4. **Verificar Despliegue Final:** Vuelva a la pestaña **'actions'** en GitHub. El proceso de *build* y *deploy* se iniciará nuevamente. Una vez que se complete, la publicación y los cambios serán visibles en su página de GitHub.

## 2. Guía Esencial de Sintaxis Markdown para Posts de Jekyll

Este tutorial cubre la sintaxis de Markdown necesaria para escribir el contenido de sus posts, usando como base la estructura requerida por Jekyll y el tema Cheri Starter.

### 1. Estructura y Metadatos Requeridos

Antes de escribir el contenido, cada archivo de post debe seguir la convención de nombre y debe incluir el bloque de metadatos (Front Matter) [1].

#### Convención de Nombres de Archivo
El archivo debe estar nombrado con la fecha y el nombre del post, separados por guiones [1]:

`YYYY-MM-DD-nombre-del-post.md`

*Ejemplo:* `2024-06-03-hello-world-post.md` [1].

#### Bloque de Metadatos (Front Matter)
Este bloque debe estar al inicio y encerrado entre tres guiones (`---`). Es obligatorio incluir `title`, `date`, `categories` y `tags` [1].

```markdown
---
title: Mi Primera Publicación
date: 2024-06-03
categories: [Programación, Web]
tags: [jekyll, markdown, github]
---
```

### 2. Encabezados (Headers)

Los encabezados se utilizan para jerarquizar el contenido de su texto. Se definen usando el símbolo de almohadilla (#).
```markdown
# Encabezado Nivel 1 (H1)
## Encabezado Nivel 2 (H2)
### Encabezado Nivel 3 (H3)
#### Encabezado Nivel 4 (H4)
```
(El contenido del post comienza después del Front Matter. Las fuentes mencionan específicamente la adición de un "header".)

### 3. Estilos de Texto

| Estilo                  | Sintaxis Markdown                               |
| :---------------------- | :---------------------------------------------- |
| **Negrita**             | `**Texto en negrita**` o `__Texto en negrita__` |
| *Cursiva*               | `*Texto en cursiva*` o `_Texto en cursiva_`     |
| **Negrita y *Cursiva*** | `***Texto combinado***`                         |
| ~~Tachado~~             | `~~Texto tachado~~`                             |


### 4. Párrafos y Saltos de Línea
Los párrafos se crean dejando una línea en blanco entre bloques de texto.

```markdown
Este es un párrafo.

Este es otro párrafo.
```

### 5. Listas

Listas Ordenadas (Numéricas)
```markdown
1. Primer ítem
2. Segundo ítem
3. Tercer ítem
```

Listas No Ordenadas (Viñetas)
```markdown
* Elemento uno
* Elemento dos
    * Sub-elemento (debe estar indentado)
```

### 6. Enlaces e Imágenes

**Enlaces**

Se usa el texto del enlace entre corchetes, seguido de la URL entre paréntesis.

```markdown
[Mi sitio web personal](https://mi-sitio.com)
```

**Imágenes**

Se utiliza la misma sintaxis de enlace, pero se antepone un signo de exclamación (!).

```markdown
![Descripción de la imagen](URL de la imagen)
```

### 7. Citas en Bloque (Blockquotes)

Se utilizan para citar texto externo o resaltar un pasaje. Se definen usando el símbolo de mayor que (>).

```markdown
> "La clave del éxito es la paciencia."
>
> — Cita anónima
```

### 8. Código

**Código en Línea (Inline Code)**

Para referencias cortas a código dentro de una línea de texto.

```markdown
`Envuelva el código con comillas inversas (backticks).`
```

**Bloques de Código (Fenced Code Blocks)**

Para mostrar bloques de código más largos, útil para ejemplos de lenguajes de programación. Se usan tres backticks al inicio y al final. Se recomienda especificar el lenguaje para el resaltado de sintaxis.


````markdown
```ruby
def saludo
  puts "Hola Jekyll!"
end
```
````

### 9. Línea Horizontal

Útil para crear una división temática o visual entre secciones.

```markdown
***

o

---
```



### 10. Sintaxis de Tabla

Las tablas utilizan barras verticales (`|`) para delimitar las columnas y guiones (-) para separar el encabezado del cuerpo.

```markdown
| Nombre de la Columna |  Descripción del Dato  |  Alineación (Ejemplo) |
| :------------------- | :--------------------: | --------------------: |
| Ítem A               | Descripción del ítem A | Alineado a la derecha |
| Ítem B               | Descripción del ítem B |    Alineado al centro |
```

*   **Alineación:** La línea de guiones define la alineación del texto en cada columna.
    *   `:---` (Izquierda)
    *   `:---:` (Centro)
    *   `---:` (Derecha)

### 11. Sintaxis para Enlazar un Archivo PDF

La sintaxis es la misma que la de un enlace estándar. Es crucial que el archivo PDF esté cargado en su repositorio de Jekyll (por ejemplo, en una carpeta llamada `/assets/`) para que funcione en GitHub Pages.

```markdown
[Texto que verán los usuarios (Ej: Descargar PDF)](/assets/nombre-de-su-archivo.pdf)
```

*Ejemplo práctico asumiendo que el PDF está en la carpeta `assets`.*

## 3. ¿Qué hemos construido? El framework detrás del blog

Cuando seguiste los pasos de la Sección 1 — crear el repositorio, configurar la URL y hacer el primer commit — pusiste en marcha un sistema completo de publicación web. Vale la pena entender qué hay detrás.

### Las piezas del sistema

El blog no es un sistema único sino **cuatro tecnologías que trabajan en cadena**:

```
Markdown (.md) → Jekyll → HTML/CSS/JS → GitHub Pages
                               ↑
                         Chirpy Theme
```

| Componente | Rol | ¿Dónde vive? |
|:-----------|:----|:-------------|
| **Markdown** | Lenguaje con el que escribes el contenido | Archivos `.md` en `_posts/` |
| **Jekyll** | Motor que convierte Markdown en HTML estático | Corre durante el build |
| **Chirpy** | Tema visual: layout, estilos, navegación, modo oscuro | `_config.yml` + gems |
| **GitHub Pages** | Servidor web que sirve los archivos HTML al mundo | Infraestructura de GitHub |
| **GitHub Actions** | Orquestador que ejecuta todo el proceso automáticamente | `.github/workflows/` |

### El flujo completo: del commit al sitio publicado

Cada vez que haces `git push`, esto es lo que ocurre:

```
1. git push  →  GitHub recibe el commit
                      ↓
2. GitHub Actions detecta el cambio y lanza el workflow
                      ↓
3. Jekyll build: lee _posts/*.md + _config.yml + Chirpy
                 genera carpeta _site/ con HTML puro
                      ↓
4. GitHub Pages publica _site/ en username.github.io
                      ↓
5. El visitante abre el browser → recibe HTML estático
```

> **Punto clave:** el visitante **nunca ejecuta Ruby ni Jekyll**. Solo recibe archivos HTML/CSS/JS estáticos. Ruby y Jekyll solo corren durante el build — en GitHub Actions o en tu máquina local.

### ¿Qué es un sitio estático?

A diferencia de WordPress o una aplicación web tradicional que genera páginas *en el momento* en que el usuario las pide (consultas a base de datos, lógica de servidor), un sitio estático **pre-genera todo el HTML de antemano**. Resultado:

- **Velocidad:** no hay tiempo de respuesta de servidor ni base de datos
- **Seguridad:** no hay superficie de ataque en el servidor (no hay PHP, SQL, sessions)
- **Costo:** GitHub Pages es completamente gratuito para sitios estáticos públicos
- **Limitación:** el contenido no cambia en tiempo real sin un nuevo build

### ¿Qué hace GitHub Actions exactamente?

El archivo `.github/workflows/pages-deploy.yml` (incluido en el template de Chirpy) define el pipeline de CI/CD del blog:

```yaml
# Simplificado — lo que hace el workflow:
on: [push]          # Se activa con cada push a main
jobs:
  build:
    - checkout      # Descarga el código
    - ruby setup    # Instala Ruby + bundler
    - bundle install # Instala Jekyll + Chirpy + plugins
    - jekyll build  # Genera _site/
  deploy:
    - upload _site/ → GitHub Pages
```

Esto es exactamente lo mismo que haces con `bundle exec jekyll serve` en tu máquina local — la diferencia es que GitHub Actions lo ejecuta en la nube cada vez que haces push, sin que tengas que hacer nada manualmente.

### ¿Por qué esto importa en ciberseguridad?

Este stack — **código fuente → pipeline automatizado → artefacto desplegado** — es el patrón estándar de CI/CD que encontrarás en cualquier empresa de software. Entender cómo funciona es fundamental porque:

- Los pipelines CI/CD son vectores de ataque frecuentes (**supply chain attacks**)
- Una credencial comprometida en GitHub Actions puede llevar a despliegues maliciosos
- El concepto de "superficie de ataque mínima" del sitio estático es un principio de diseño seguro aplicable a cualquier sistema
