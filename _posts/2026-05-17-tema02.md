## 2. Networking en VirtualBox

Para que el laboratorio funcione, las dos máquinas virtuales — el **atacante** (Kali Linux) y la **víctima** (Metasploitable 3) — deben poder comunicarse entre sí en una red controlada. VirtualBox ofrece varios modos de red, cada uno con características distintas.

### ¿Qué es un adaptador de red virtual?

Cuando VirtualBox crea una VM, le asigna una tarjeta de red virtual (NIC). Esta NIC puede conectarse a distintos tipos de redes virtuales según el modo que se configure. El host (tu PC) actúa como el "router" o "switch" virtual dependiendo del modo elegido.

### Modos de red disponibles

#### NAT (Network Address Translation)

```
Internet ←→ [Host] ←→ [VM]
```

- La VM tiene acceso a internet a través del host.
- La VM **no puede ser alcanzada** desde otras VMs ni desde el host directamente.
- El host hace de router: la VM recibe una IP privada (ej. `10.0.2.15`) que el host traduce.
- **Problema para el lab:** Kali no puede "ver" a Metasploitable y viceversa.

> **Uso típico:** cuando solo necesitas internet en la VM y nada más.

---

#### NAT Network

```
Internet ←→ [Host] ←→ [Switch virtual NAT] ←→ [Kali] + [Metasploitable]
```

- Similar a NAT, pero las VMs **sí pueden comunicarse entre sí**.
- Ambas VMs comparten una red NAT interna (ej. `10.0.2.0/24`).
- Todas las VMs del grupo tienen acceso a internet.
- Requiere crear la red NAT primero en: *Archivo → Herramientas → Network Manager → NAT Networks*.

> **Uso:** lab con acceso a internet + comunicación entre VMs.

---

#### Host-Only Adapter

```
[Host] ←→ [Switch virtual aislado] ←→ [Kali] + [Metasploitable]
         (sin internet)
```

- Las VMs se comunican entre sí **y con el host**, pero **no tienen acceso a internet**.
- VirtualBox crea un adaptador virtual en el host (ej. `vboxnet0`) con su propio rango IP (ej. `192.168.56.0/24`).
- La red está completamente aislada del mundo exterior.

> **Uso recomendado para este lab.** Es la opción más segura para practicar ataques: Metasploitable no tiene salida a internet, no puede ser explotada desde afuera, y Kali puede atacarla libremente sin riesgo.

---

#### Internal Network

```
[Kali] ←→ [Switch virtual completamente aislado] ←→ [Metasploitable]
         (sin internet, sin host)
```

- Las VMs solo se comunican entre sí. **Ni el host ni internet tienen acceso.**
- La red no existe fuera de VirtualBox — máximo aislamiento posible.
- Requiere configurar IPs manualmente en cada VM (no hay DHCP por defecto).

> **Uso:** entornos de máximo aislamiento, p. ej. análisis de malware.

---

#### Bridged Adapter

```
[Router físico / Red corporativa] ←→ [Host] ←→ [VM como dispositivo independiente]
```

- La VM aparece como un dispositivo físico más en la red real.
- Recibe IP del router real (DHCP de tu casa/universidad).
- Cualquier dispositivo en la red puede ver la VM y viceversa.

> **Peligro para este lab:** Metasploitable quedaría expuesta en la red de la universidad. **No usar en entornos compartidos.**

---

### Comparativa de modos

| Modo | Kali → Metasploitable | Acceso a Internet | Host → VMs | Aislamiento |
|:-----|:---------------------:|:-----------------:|:----------:|:-----------:|
| NAT | ✗ | ✓ | ✗ | Medio |
| NAT Network | ✓ | ✓ | ✗ | Medio |
| Host-Only | ✓ | ✗ | ✓ | Alto |
| Internal | ✓ | ✗ | ✗ | Máximo |
| Bridged | ✓ | ✓ | ✓ | Ninguno |

### Configuración recomendada para este laboratorio

Usaremos **NAT Network Adapter** en ambas VMs:

1. En VirtualBox, ir a *File → Tools → Network Manager* y verificar que existe `NatNetwork` (rango `10.0.2.0/24`).
2. Para cada VM (Kali y Metasploitable): *Configuración → Red → Adaptador 1 → Nat Network → NatNetwork*.
3. Iniciar ambas VMs y verificar conectividad:

```bash
# En Kali — verificar su IP
ip a

# En Kali — verificar que Metasploitable responde
ping 10.0.2.X   # IP asignada a Metasploitable
```

Con esto, Kali puede atacar a Metasploitable en una red privada y controlada, sin riesgo de exponer la máquina vulnerable al exterior.

## 3. The Kill Chain 

¿Qué es el Kill Chain? [(Link)](https://youtu.be/oCUrkc_0tmw?si=bypiWhaGtWzEOrIQ)

### Reconnaissance

**Objetivo**: Recolección de información sobre el sistema objetivo para encontrar debilidades en la red o sistema

Herramientas para el reconocimiento Pasivo:
- Whois
- Arin
- Google
- Shodan
- Job listings
- Company website

Medidas de protección contra reconocimiento pasivo:

- Limitar la información pública
- Uso moderado de RRSS
- Modificar los mensajes de error de servicios que corren en el server

Herramientas para el reconocimiento activo:

- Nmap
- Port scanning
- Banner grabbing
- Vulnerability

Medidas de protección contra reconocimiento activo:

- Desactivar servicios/puertos que no se usen
- Honeypots
- Firewall
- IPS
- Bloqueo de tráfico entrante proveniente de Tor & VPNs de terceros

### Weaponization

**Objetivo**: crear el ataque para explotar las debilidades encontradas 

Herramientas de ataque:

- Metasploit
- Exploit-DB
- Veil Framework
- Social Engineering Toolkit
- Cain and Abel
- SQLMap
- Aircrack
- Maltego Web app
- Burp Suite
- FratRAT

Herramientas de defensa:

- Patch management
- Deshabilitación
    - Office Macros
    - Javascript
    - Browser plugins
- Antivirus
- IPS
- Email security
- MFA
- Audit logging

### Delivery

**Objetivo**: entregar del exploit de la vulnerabilidad detectada.

Algunas herramientas:

- Websites
- Social media
- User input
- Email
- USB

Medidas de protección:

- Web filtering / DMS Filtering
- Phishing campaigns
- IPS/IDS
- DKIM / SPF
- Disabling USB

### Exploitation

**Objetivo**: ejecución del ataque y ganar acceso al sistema target.

Algunas vectores de ataque:

- Buffer overflow
- SQL Injection
- Malware
- Client-side exploit (Javascript Hijack)

Medidas de protección:

- Data execution prevention (DEP)
- Anti-exploit
- Sandbox

### Installation

**Objetivo**: inyectar un payload, o algún ejecutable, luego de la etapa de exploitation, para ganar un acceso con mayores privilegios.

Herramientas:

- DLL Hijacking
- Meterpreter
- Remote Access Tools (RAT)
- Registry changes
- Powershell commands

Medidas de protección:

- Protección limitada
    - Linux: CHROOT Jail
    - Windows: Disable Powershell
- Detección:
    - UBA / EDR
- Respuesta:
    - Seguimiento a incidencias (Device > Aislamiento > Limpieza)
- Recovering:
    - Restoration

### Command and Control

**Objetivo**: ganar un control **remoto** del sistema target.

Medidas de protección:

- Network Segmentation
- Micro Segmentation
- NGFW: C&C Blocking
- DNS Redirect (DBA, Fast Flux)
-Application control (SSH, Telnet, Netcat, RDP / SSL packet -inspection)
- IoC (observed behavior)

### Post-exploitation

**Objetivo**: el atacante ejecuta la acción de su objetivo final.

- Entender las motivaciones del atacante.
- La mayoría de las veces el objetivo final es **exfiltrar data**.
- Algunas medidas de protección:
    - Data leakage prevention (DLP)
    - User behavior analysis (UBA)
- Otras veces, el objetivo de la primera etapa de un ataque es ganar lo que se denomina como LATERAL MOVEMENT, es decir, la capacidad de moverse hacia otros elementos de la red.

- Para prevenir ello, es importante la implementación de la segmentación de red. -->
