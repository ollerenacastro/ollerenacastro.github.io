---
Jenkins es, en esencia, **el GitHub Actions de antes** — hace exactamente lo mismo que GitHub Actions, GitLab CI o CircleCI, pero instalado en un servidor propio de la empresa (on-premise) en lugar de en la nube. Si alguna vez hiciste un `git push` y viste un ícono verde de "checks passed" en GitHub, ya interactuaste con un sistema CI/CD. Jenkins fue el pionero que estandarizó ese flujo en la industria.

El flujo de trabajo típico es:
1. Un desarrollador hace `git push` con nuevo código
2. Jenkins detecta el cambio automáticamente (via webhook)
3. Ejecuta las pruebas automáticas (`npm test`, `pytest`, `mvn test`)
4. Compila y empaqueta el artefacto (genera el `.jar`, la imagen Docker, el `.exe`)
5. Despliega en el servidor de producción vía SSH

Para ejecutar los pasos 4 y 5, Jenkins necesita acceso privilegiado a los servidores de destino. De ahí que acumule credenciales sensibles.

Si Jenkins es el robot que automatiza el pipeline, **Groovy es el lenguaje en que le das instrucciones a ese robot**. La Script Console es como abrir la terminal del robot y escribirle comandos directamente. Groovy es a Java lo que Python es a C — más conciso y fácil de escribir, pero con acceso completo al mismo poder de la JVM.
---

#### Prerequisito: acceso a la red interna

A diferencia de EternalBlue (que puede ejecutarse desde internet si el puerto 445 está expuesto), Jenkins en la mayoría de las organizaciones **no es accesible desde el exterior** — vive en la red interna. Esto significa que este Kill Chain asume que el adversario ya tiene acceso a la red interna del target. Ese acceso previo puede provenir de:

- **Lateral movement**: el atacante comprometió primero otra máquina de la red (ej. vía EternalBlue) y desde ahí pivota
- **VPN comprometida**: credenciales de VPN obtenidas por phishing o brute force
- **Insider threat**: un empleado malintencionado ya dentro de la red

En nuestro lab, este prerrequisito está implícito: nuestra VM Kali está en la misma NAT Network que Metasploitable3, simulando que ya tenemos presencia en la red interna.

Existe también el escenario de Jenkins expuesto directamente a internet (el que generó las 50,000 instancias en Shodan), pero el caso más común — y el más enseñable — es el del servidor interno mal configurado.
